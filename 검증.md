# 검증 가이드

ControlBit UWB 앱이 의도대로 동작하는지 단계별로 검증하는 방법.
각 단계는 **앞 단계가 통과해야** 의미가 있다 — 위에서 아래로 진행.

---

## 0. 준비물

| 항목         | 요구                                                                |
| ------------ | ------------------------------------------------------------------- |
| 폰           | UWB 하드웨어 탑재 Android (Pixel 6 Pro+, S21 Ultra+ 등), Android 13+ |
| 안커         | maUWB_DW3000 (커스텀 펌웨어 — 본 프로젝트의 BLE OOB 사양 구현)       |
| 호스트       | Flutter 3.41+, Android Studio (또는 `adb` + Android SDK 36)          |
| 케이블       | USB-C (디버깅), 가능하면 안커 시리얼 모니터링용 추가 포트            |

> Android 시스템 설정 → 연결 → UWB 가 켜져 있는지 확인. 일부 기기는 토글이 숨겨져 있어 `pm has-feature android.hardware.uwb` 로 확인.

## 1. 정적 검증 — 빌드와 단위 테스트

### 1-1. 의존성/코드 생성

```bash
flutter pub get
dart run build_runner build --delete-conflicting-outputs
```

기대: `lib/models/*.freezed.dart`, `lib/models/*.g.dart` 생성.
실패 시: freezed/json_serializable 버전 충돌 — `pubspec.lock` 삭제 후 재시도.

### 1-2. 정적 분석

```bash
flutter analyze
```

기대: `No issues found!`

### 1-3. 단위 테스트 (BLE 와이어 포맷)

```bash
flutter test
```

검증 대상: `test/ble_codec_test.dart`
- `SessionConfig`(32 B) 인코딩 — protocolVersion / sessionId LE / sessionKey 오프셋 / updateRateType / 예약 영역
- `AnchorInfo`(16 B) 디코딩 — channel mask 비트 검사, vendorId 7 B
- `SessionState`(4 B) 디코딩 — state/err/uptime

**왜 중요한가**: 이 테스트가 실패하면 폰과 안커 펌웨어가 같은 바이트를 보고도 다르게 해석한다는 뜻. 무선 디버깅이 거의 불가능해진다.

### 1-4. Android 네이티브 빌드

```bash
flutter build apk --debug
```

기대: `build/app/outputs/flutter-apk/app-debug.apk` 생성, 경고는 있어도 OK.
실패 시 자주 보는 원인:
- `compileSdk` < 36: `androidx.core.uwb:1.0.0-rc01`이 36 요구
- AGP/Gradle 버전 미달: `android/gradle/wrapper/gradle-wrapper.properties`에서 8.11.1+ 확인

## 2. 디바이스 검증 — UWB 가용성

### 2-1. APK 설치 및 권한

```bash
flutter install
adb shell pm grant com.controlbituwb.app_uwb android.permission.UWB_RANGING
adb shell pm grant com.controlbituwb.app_uwb android.permission.BLUETOOTH_SCAN
adb shell pm grant com.controlbituwb.app_uwb android.permission.BLUETOOTH_CONNECT
```

> 앱 첫 실행 시 권한 다이얼로그가 뜨면 모두 허용. 위 명령은 다이얼로그를 건너뛰고 싶을 때만.

### 2-2. UWB 하드웨어 인식

```bash
adb shell pm has-feature android.hardware.uwb
# → true 여야 함
```

앱 로그:
```bash
adb logcat | grep -i uwb
```
앱 실행 직후 `UwbBridge.isUwbAvailable() == true` 가 흐름 어딘가에 나와야 한다(없으면 `lib/services/uwb_service.dart`에 `print` 한 줄 임시 추가).

### 2-3. 로컬 주소/Capabilities

스캔 화면 진입 시 (또는 디버거에서 직접 호출):

```dart
final caps = await UwbPlatform.instance.getRangingCapabilities();
final addr = await UwbPlatform.instance.getLocalAddress();
print('caps=$caps, localAddr=${addr.toList()}');
```

기대:
- `addr.length == 2` (UWB short address 2 B)
- `caps.isDistanceSupported == true`
- `caps.supportedChannels`가 비어있지 않음 (보통 `[5, 9]`)

## 3. BLE OOB 검증 — 안커와의 핸드셰이크

### 3-1. 광고 발견

`scan_screen` 진입 → 안커가 `BleUuids.service`(`8E7A0001-…`)를 광고하는지 확인.
폰에서 안 잡히면 안커 측에서 보낸 패킷을 nRF Connect 같은 외부 도구로 먼저 확인:
- Service UUID 노출되는가?
- TX power, advertising interval 적절한가?

### 3-2. GATT 연결과 MTU

연결 직후 로그에서 다음을 확인:
- ATT MTU 협상 결과 ≥ 100 (`BleUuids.negotiateMtu`)
- 5개 캐릭터리스틱 모두 발견 (Service / AnchorInfo / SessionConfig / SessionState / PhoneAddress)

`flutter_blue_plus`가 MTU 100을 거부할 경우: 일부 안드로이드 BLE 스택은 23 → 247 사이만 지원. 안커 펌웨어에서 247을 받아주는지 먼저 확인.

### 3-3. AnchorInfo 읽기

| 검사 항목                | 기대                                       |
| ------------------------ | ------------------------------------------ |
| `protocolVersion`        | `0x01` (앱과 동일)                         |
| `anchorShortAddr.length` | 2                                          |
| `supportedChannelMask`   | bit5 또는 bit9 set                         |
| `mtuPreference`          | 협상된 MTU 이상 요청 안 함                 |

다르면 펌웨어와 앱의 `BleCodec`이 같은 16 B 레이아웃을 쓰는지부터 재확인.

### 3-4. PhoneAddress 쓰기 → SessionConfig 쓰기

순서 (`session_orchestrator.dart` 참고):
1. 폰 → 안커: `PhoneAddress` (2 B, 폰의 UWB short address)
2. 폰 → 안커: `SessionConfig` (32 B)
3. 안커 → 폰: `SessionState` notify가 `idle → armed → active` 순으로 흘러야 함

`armed`에서 멈추면: 펌웨어가 ConfigId / channel / preamble을 거부한 상태. `lastErrorCode` 확인.
`error`로 떨어지면: `lastErrorCode`로 펌웨어 원인 코드 매핑 (펌웨어 문서 필요).

## 4. UWB 레인징 검증

### 4-1. 첫 거리값 수신

`SessionState = active` 진입 후 1~2초 안에 EventChannel로 `RESULT` 이벤트가 와야 한다.

`adb logcat`에서:
```bash
adb logcat | grep -E 'RESULT|distanceM'
```

기대 로그 예:
```
{type: RESULT, distanceM: 1.23, azimuthDeg: -12.5, elevationDeg: 0.0, ...}
```

### 4-2. 정확도 검증 (실측 비교)

| 거리        | 기대 오차              |
| ----------- | ---------------------- |
| 0.5 m       | ±10 cm                 |
| 1 m         | ±10 cm                 |
| 3 m         | ±15 cm                 |
| 5 m         | ±20 cm                 |
| 10 m (LOS)  | ±30 cm                 |

벽 너머/금속 가림(NLOS)에서는 큰 폭으로 튀는 것이 정상 — 거부할 게 아니라 표시상 신뢰도를 낮춰야 한다.

### 4-3. AoA(있을 경우)

`caps.isAzimuthalAngleSupported == true`인 기기만:
- 안커를 폰 앞에 두고 좌/우로 이동시키며 `azimuthDeg` 부호가 ±로 갈리는지
- compass 위젯이 안커 방향을 가리키는지

### 4-4. 끊김/재연결

- 안커 전원 OFF → 앱이 `PEER_DISCONNECTED` 이벤트를 받고 화면이 **error 상태로 전이**해야 한다 (멈춤/크래시 ❌)
- 다시 ON → scan 화면으로 돌아가 재연결 가능해야 함

### 4-5. 백그라운드/포그라운드

- 앱 → 백그라운드 → 5초 → 포그라운드: 세션 재개 또는 깔끔한 종료
- 화면 회전: 세션이 죽지 않아야 (Riverpod provider 라이프사이클 점검)

## 6. 흔한 실패 패턴

| 증상                                  | 1순위 의심                                         |
| ------------------------------------- | -------------------------------------------------- |
| `isUwbAvailable() == false`            | 시스템 설정에서 UWB OFF, 또는 비-UWB 기기          |
| GATT 연결은 되는데 AnchorInfo 길이 ≠16 | 펌웨어와 앱의 구조체 패딩 불일치                   |
| SessionState가 `armed`에서 멈춤         | 펌웨어가 channel/preamble/configId 미지원          |
| 거리값이 NaN/0                         | RangingParameters의 sessionKey/sessionId 양쪽 동일성 깨짐 |
| 첫 RESULT까지 5 s 이상                  | rangingInterval / updateRateType 너무 느림         |
| 거리만 나오고 AoA = null                 | 기기가 AoA 미지원 (capabilities에서 미리 확인)     |
